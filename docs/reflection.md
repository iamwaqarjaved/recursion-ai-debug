## When Can You Trust an AI's Recursion Trace?

After running all five components of this assignment, the answer is nuanced: AI is a fast, reliable *first-pass tool* for recursion, but it degrades silently in ways that are hard to detect without running the code.

### Where AI performed well

**Bug identification was the strongest result.** All three structural bugs — missing base case, unreachable base case via float division, and wrong-direction parameter — were correctly identified on the first prompt. The AI named the exact line causing the problem and explained the mechanism clearly. This is AI at its best: pattern matching against a well-understood class of errors.

**Fix proposals were syntactically and logically correct** for all three cases. Each suggested fix ran cleanly and produced the expected output. That said, I still ran each fix before trusting it — the AI could have introduced a subtle off-by-one or type error I wouldn't catch by reading alone.

**Shallow linear traces were accurate.** The `tree_sum` trace at depth 4 matched the print-verified output exactly: call order, intermediate values, and final results were all correct. For linear recursion under about 10 frames, the AI's mental model of the call stack is reliable.

### Where AI failed

**The most important failure was over-generalization on Bug C.** The AI correctly diagnosed `exp + 1` as moving away from the base case for positive inputs, then incorrectly concluded this applied to *all* inputs. It missed that negative starting values cause `exp + 1` to count *toward* zero, making the base case reachable. The function terminates normally for `power(2, -3)` — a fact the AI would have discovered instantly with one test run, but didn't notice by static analysis alone.

**The AI omitted evaluation-order details** in the `tree_sum` trace. It didn't note that `sum(tree_sum(child) for child in node)` uses a lazy generator — children are evaluated as `sum` pulls them, not all at once. This is irrelevant for correctness in this case but would matter in code with side effects or mutating shared state.

### At what depth does AI lose track?

For *linear* recursion (one recursive call per frame), accuracy held up through depth ~15 in informal testing. For *branching* recursion (Fibonacci, binary tree traversal with two calls per frame), degradation appeared around depth 5–6 because the number of live frames grows exponentially. The AI cannot hold 2^n frames in working context the way it holds n frames.

### The core principle

An AI trace is a prediction, not an observation. It is generated by pattern-matching against training examples of similar recursive functions — not by simulating the actual Python interpreter. For any recursion deeper than what a human could trace on paper in five minutes, or for any function with non-obvious input behavior, the only reliable verification method is running the code with print statements or a debugger.

The discipline this assignment builds is not "use AI vs. don't use AI." It is: **use AI to accelerate the first pass, then verify with execution before trusting the result.** The two tools are complementary. The AI gets you to a hypothesis in seconds; the interpreter confirms or disproves it in milliseconds.
